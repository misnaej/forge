## Forge-specific rules

- **Version derivation**: pip package version comes from the latest git tag via setuptools-scm. There is no manual `version = "x.y.z"` in `pyproject.toml`. Release flow: `git tag vX.Y.Z && git push origin vX.Y.Z`.
- **PR & branch granularity â€” bundle cohesive work, don't reflex-split.** A PR is the smallest *cohesive* unit, not the smallest possible one. In forge every PR also costs a rolling-next `plugin.json` bump + a `dev` tag (and, at minor boundaries, a promotion), so splitting *related* changes across PRs multiplies that ceremony to buy isolation nobody needs. **Default to ONE PR** when the changes share a subsystem, must land together, or one half is meaningless without the other â€” e.g. a bug a type checker surfaces *while you build the feature that runs it* belongs in that feature's PR, not a parallel one; a follow-up cleanup to code this PR introduces belongs here too. **Split only for a concrete reason:** independent risk/rollback, different reviewers or release cadence, a hard blocking dependency, or a diff too large to review in one sitting. "It's a separate concern" does **not** justify a separate PR when the concerns ship together â€” separateness of *topic* â‰  separateness of *delivery*. When unsure, bundle. (File a separate *issue* freely for deferred work â€” that's cheap; it's separate *PRs/branches* that add the burden.)
- **Release process â€” versioning, tag cadence, and `dev â†’ main` promotion** are specified in **[`docs/release-process.md`](docs/release-process.md)**, the single source of truth + the **invariantâ†’test contract**. Do not restate its mechanics elsewhere; point here. Operational summary:
  - `.claude-plugin/plugin.json["version"]` is **rolling-next** â€” always names the version about to be released; `step_plugin_version` enforces `plugin.json > latest_tag` on every commit (skipped when `HEAD` reproduces a tagged release's tree). After tagging `vX.Y.Z`, the next PR must bump it.
  - Tag `dev` `vX.Y.Z` after **every** merge to dev (`forge-next-prep --tag`, run by `/next` â€” **never drop `--tag`**); tag `main` **only** at the minor-boundary promotion. `@dev` gets every version, `@main` minors only.
  - Promote `dev â†’ main` **one minor at a time** via a `release/vX.Y.Z` branch (never `--head dev`) â€” the **`/promote`** skill (`.claude/skills/promote/SKILL.md`, forge-only, not shipped via the plugin).
  - **Every versioning/promotion invariant names an enforcing test in the spec.** Do NOT change that code without the matching spec test staying green â€” e.g. the guard skips on *any* tagged-release tree (not just the latest), required for staged catch-up, locked by `test_main_skips_when_head_reproduces_older_tag`. This contract exists because #43 silently broke that behavior with no test to catch it.
- **Semver policy for the plugin bump** â€” when the post-tag rolling-next bump PR lands (see [`docs/release-process.md`](docs/release-process.md) Â§1), choose the increment deliberately, not reflexively. The plugin's public surface is: every CLI in `[project.scripts]`, every agent name + canonical Output shape under `agents/`, every Claude Code hook under `claude-hooks/`, every skill under `skills/`, every FOUNDATION rule a consumer can rely on.
  - **PATCH (Z+1)** â€” internal-only changes: bug fix in an existing CLI / agent / hook with no behavior change visible to consumers; refactor with identical externals; doc typo fix; non-blocking new audit check; CLAUDE.md edits that affect only forge contributors.
  - **MINOR (Y+1, Zâ†’0)** â€” additive, backward-compatible: new CLI, new agent, new hook, new skill, new mandatory rule in FOUNDATION or `_TEMPLATE.md` that does NOT break existing consumer artifacts (e.g. a new reporter-agent header that older agents simply lack â€” audit flags it but does not refuse a commit), new mode/flag on an existing CLI or agent, new pre-commit step that self-skips when not opted in.
  - **MAJOR (X+1, Yâ†’0, Zâ†’0)** â€” breaking: CLI renamed / removed / argument-incompatible; agent renamed / removed / Output shape changed in a way that breaks existing parsers (e.g. `pr-manager`); hook semantics inverted (allow â†’ block); FOUNDATION rule promoted from soft to blocking; manifest layout change. Any consumer upgrade that requires action beyond `forge-upgrade` is MAJOR.
  - **When in doubt** between minor and patch: lean minor. Tags are free; misleading patch bumps obscure real changes from consumers reading `git log v1.3.4..v1.3.5`.
- **`dev/` is forge-only â€” `./dev/setup.sh` is the ONLY install/refresh command**: `dev/setup.sh` (conda env, editable install, `install-forge-bootstrap`, doctor) is forge's own bootstrap and is **re-run safe (idempotent)**. **To refresh the editable install â€” e.g. to pick up a new `[project.scripts]` CLI / entry point after pulling a PR â€” re-run `./dev/setup.sh`.** **NEVER tell the user to run (or run) a raw `pip install`** â€” per FOUNDATION Â§2 the *exact command* for forge is this bootstrap script, not bare pip. (To automate the refresh after every pull, drop a script that runs `./dev/setup.sh` into `.githooks/post-merge.d/` â€” the consumer extension point added in #10.) Not a consumer pattern; consumers use whatever Python env tool they prefer + `install-forge-githooks`.
- **Plugin layout**: `.claude-plugin/` (manifest), `agents/`, `skills/` (auto-discovered), `claude-hooks/` (referenced via `${CLAUDE_PLUGIN_ROOT}/claude-hooks/...` in `plugin.json`). Do not confuse with `.githooks/` (git hooks, separate domain).
- **Git-hook version sidecar**: the tracked `.githooks/*` wrappers are **version-free**. The forge version that last wrote them lives in the gitignored `.githooks/.forge-hook-version` sidecar, read at runtime by each hook's staleness preamble (`"$(dirname "$0")/.forge-hook-version"`). This keeps the committed hooks byte-stable across version bumps â€” critical here because forge is editable-installed, so `metadata.version()` returns an ever-changing `X.Y.Z.devN+g<sha>.d<date>` string that would otherwise re-stamp (and dirty) the hooks on every `post-merge`/`post-checkout` auto-refresh. `install-forge-githooks` writes the sidecar and ensures it's gitignored on every run; the marker carries only `body-sha=` (consumer-edit detection), never `forge-version=`.
- **FOUNDATION.md location**: canonical file at repo root for contributors. Shipped via pip as `src/forge/data/FOUNDATION.md` (symlink). `install-forge-claude-md` reads from `importlib.resources`.
- **`manifest_json` step** in `forge-precommit` fires here because forge ships a plugin (`.claude-plugin/` present). It self-skips in consumer repos that don't ship a plugin.
- **`pip_audit` step is non-blocking by default** (since v1.1.3): renders as yellow `WARN` if CVEs are found, never fails the overall pre-commit. Use `StepResult(non_blocking=True)` for any future advisory step. **Configurable since v2.3.0**: `[tool.forge.pip_audit].blocking = true` makes CVE findings a hard `FAIL` (same opt-in `_forge_step_config(...).get("blocking", False)` pattern as `typecheck`/`doctest`); a missing `pip-audit` binary always stays a non-blocking WARN (broken-install signal, not a finding). Forge keeps its own pip_audit **non-blocking** â€” an upstream dependency CVE (not forge's code, often unfixable same-day) should warn, not halt every commit â€” while running **`typecheck` blocking** (`[tool.forge.typecheck].blocking = true`): type errors are forge's own code.
- **`docstring_coverage` step is non-blocking**: mirrors the `pip_audit` pattern â€” `StepResult(non_blocking=True)`. See FOUNDATION Â§8 "Docstring coverage â€” three layered enforcers" for the full rationale and the `MISSING: <path>:<line>:<name>` dispatch contract with `forge:precommit-fixer`.
- **CLAUDE.md is consumer-owned** (since v1.1.3): foundation lives in `FOUNDATION.md`, included via `@FOUNDATION.md` near the top of this file. `install-forge-claude-md` only writes/updates `FOUNDATION.md` and the initial CLAUDE.md scaffold. Once scaffolded, this file is forge-developer territory.
- **Logging convention** (forge's documented equivalent of FOUNDATION Â§9): forge has no `common.logging` module. Forge CLIs use `from forge.git_utils import configure_cli_logging` once at module load + `logger = logging.getLogger(__name__)` for the module logger. `configure_cli_logging()` configures the root logger with `INFO` level and a bare-message formatter. Module-level `logging.getLogger(__name__)` is the deliberate pattern here, not a Â§9 violation â€” per Â§9's "consumer repos either adopt this module or document their own logging entry point in their CLAUDE.md".
- **CLI wiring is enforced by reachability** (`verify-forge-cli-wiring`, pre-commit step `cli_wiring`): every `[project.scripts]` entry in `pyproject.toml` must appear at least once in a wiring source â€” `src/forge/install_bootstrap.py` (STEPS list), `src/forge/precommit.py` (step functions), `src/forge/audit/` (audit orchestrator + steps), `.githooks/`, `claude-hooks/`, `dev/`, `agents/`, or `skills/`. The verifier greps these paths and fails the pre-commit when a script name is unreachable, excluding the CLI's own implementation file. Why: forge has many invocation paths and a new `[project.scripts]` line can land without being wired into any of them, producing an installed-but-uncalled CLI. The grep IS the contract â€” no metadata registry to keep in sync. If a CLI is intentionally unwired (release tooling, CI-only utility), add it to `cli_wiring_exempt.toml` at the repo root with `[exempt."<name>"] reason = "..."`; the verifier exempts it and reports stale entries when the script is later removed. The step self-skips when the consumer repo has no `[project.scripts]` table.
- **Approved `# noqa` exceptions** (FOUNDATION Â§5 normally permits only `E402`):
  - `# noqa: N802` â€” for `ast.NodeVisitor` `visit_*` methods, stdlib forces PascalCase suffixes like `visit_FunctionDef` (e.g. `verify_docstrings.py`).
  - `# noqa: PLC0415` â€” for guarded optional-dependency imports under `try / except ImportError` inside a helper, where moving the import to module level would break consumers without the extra (e.g. `audit/orphans.py:78` for `vulture`). Module-level guarded imports are preferred when feasible; reserve PLC0415 for cases where the optional dep is only needed inside one function.
